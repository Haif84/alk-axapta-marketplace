# alk-hooks-claude2telegram

Гейт подтверждения разрешений Claude Code через Telegram: когда VS Code показывает диалог "Allow/Deny", копия запроса с кнопками уходит в Telegram — можно ответить с телефона, гонка с локальным диалогом решается тем, кто ответит первым. Дополнительно доступна команда `/wait` у бота — временно увеличивает таймаут ожидания ответа (10 мин / 1ч / 4ч / 8ч / 12ч, на фиксированные 24 часа).

С версии 1.1.0: узнаваемый алиас машины вместо `COMPUTERNAME`, отдельная тема в Telegram-группе на каждый проект (если в группе включён режим «Темы»), полный текст каждого ответа Claude (не только финальный), и возможность написать в теме проекта из Telegram — сообщение подхватится при следующем обращении к сессии в VS Code.

С версии 1.2.0: команда `/auto` — авто-подтверждение запросов на заданный срок (диалоги в IDE не появляются, итог приходит сводкой), и все настройки бота (`/wait`, `/length`, `/auto`) теперь привязаны к вашему `chat_id`, а не общие на весь relay.

## Установка

1. `/plugin marketplace add Haif84/alk-axapta-marketplace` (если ещё не добавлен)
2. `/plugin install alk-hooks-claude2telegram@alk-axapta`
3. Создать файл `~/.claude/tg-approve.secrets.json`:
   ```json
   {
     "relay_url": "https://proxy-01.westeurope.cloudapp.azure.com:8443/notify",
     "ask_url": "https://proxy-01.westeurope.cloudapp.azure.com:8443/ask",
     "answer_url_base": "https://proxy-01.westeurope.cloudapp.azure.com:8443/answer",
     "edit_url": "https://proxy-01.westeurope.cloudapp.azure.com:8443/edit",
     "wait_budget_url": "https://proxy-01.westeurope.cloudapp.azure.com:8443/wait-budget",
     "length_budget_url": "https://proxy-01.westeurope.cloudapp.azure.com:8443/length-budget",
     "inbox_url_base": "https://proxy-01.westeurope.cloudapp.azure.com:8443/inbox",
     "auto_status_url": "https://proxy-01.westeurope.cloudapp.azure.com:8443/auto-status",
     "auto_digest_url": "https://proxy-01.westeurope.cloudapp.azure.com:8443/auto-digest",
     "score_url": "https://proxy-01.westeurope.cloudapp.azure.com:8443/score",
     "relay_secret": "<получить лично>",
     "claude_chat_id": <ваш chat_id>,
     "machine_alias": "Home"
   }
   ```
   - `relay_secret` — общий на всю команду, получить у владельца relay напрямую (не через открытый канал).
   - `claude_chat_id` — свой: создайте отдельный чат/группу, добавьте туда бота `@AINotify685_bot`, узнайте `chat_id` (например, отправив что-то боту и попросив владельца relay посмотреть логи, либо через `getUpdates`/`getChat` Bot API).
   - `machine_alias` — необязательное поле; без него в сообщениях используется `COMPUTERNAME`.
   - `length_budget_url` — необязательное поле; без него уведомления о завершении обрезаются по 3800 символов (см. «Команда /length» ниже).
   - `auto_status_url` / `auto_digest_url` — необязательные поля; без них команда `/auto` (см. ниже) просто не будет действовать на этой машине, всё остальное работает как обычно.
   - `score_url` — необязательное поле; без него не работают параноидальный режим `/auto` и команда `/score` (оценка риска). Без оценки параноидальный режим **ничего не подтверждает автоматически** — разрешения спрашиваются как обычно.

Хуки активируются сразу после установки — редактировать `settings.json` вручную не нужно.

## Темы по проектам

Каждый проект получает свою тему в Telegram-группе (создаётся автоматически при первом сообщении по этому проекту). Требует одноразовой ручной настройки группы **владельцем группы, не плагином**:
1. Включить режим «Темы» в настройках группы (Telegram превратит её в супергруппу). **Внимание:** при конвертации у группы **меняется chat_id** (старый отрицательный id → новый вида `-100…`) — обновите `claude_chat_id` в своём файле секретов, иначе все уведомления начнут молча падать с ошибкой «group chat was upgraded to a supergroup chat».
2. Выдать боту `@AINotify685_bot` право «Управление темами».

Без этого всё продолжает работать как раньше — все сообщения идут в общий поток группы (General), без тем.

## Команда /length — длина сообщения о завершении

`/length` у бота выбирает, сколько символов ответа Claude попадает в уведомление о завершении: **Минимум (500)** / **Средне (1000)** / **Максимум (3800)**. Выбор общий для всей команды, действует постоянно (переживает и рестарт relay), по умолчанию — 3800. Независимо от выбранного лимита итоговое сообщение гарантированно уложится в предел Telegram 4096 символов — даже если текст сильно «раздувается» при HTML-экранировании.

## Команда /auto — авто-подтверждение на время

`/auto` включает окно, в течение которого запросы разрешений подтверждаются автоматически:
**Выключить** / **1 час** / **4 часа** / **8 часов** / **12 часов**. По истечении окно выключается само;
перезапуск relay тоже сбрасывает его в «выключено» (в отличие от `/length`, состояние намеренно
не сохраняется на диск — это временное снятие защиты, а не долгоживущая настройка).

Пока окно активно, хук `PreToolUse` отвечает `permissionDecision: allow`, поэтому **нативный диалог
в IDE вообще не появляется** — никаких нажатий клавиш и перехвата фокуса.

Итог приходит **одной сводкой на проект**, а не сообщением на каждый вызов: `allow` на уровне
`PreToolUse` подавляет событие `PermissionRequest`, поэтому отличить «этот вызов реально спросил бы
разрешение» от «этот и так был разрешён allowlist'ом» уже невозможно — сообщение на каждый вызов
означало бы сотни сообщений в час. Сводка выглядит так и обновляется на месте:

```
🤖 Авто-режим — MyProject
Прошло без запроса: 42
Последние:
• Bash: git status
• Edit: src/app.py
```

Новое сообщение сводки начинается при включении `/auto` и после 15 минут тишины (чтобы не
редактировать сообщение, уехавшее далеко вверх по чату).

### Параноидальный режим (с 1.3.0)

`/auto` сначала спрашивает режим: **Выключить** / **Обычный** / **Параноидальный**. Обычный — всё
как описано выше. Параноидальный добавляет два шага: сначала время действия, потом **максимальный
уровень риска** для авто-подтверждения — `≤ 0 (строго)` / `≤ 20` / `≤ 40` / `≤ 60` / `≤ 80` / `≤ 99 (всё)`.

Перед каждым авто-подтверждением relay просит LLM оценить вызов по шкале 0 (безопасно) … 99 (очень
опасно) — шкалу целиком показывает команда `/default_scores`. Дальше:

- **оценка ≤ порога** — вызов проходит автоматически, строка в сводке получает оценку:
  `• 14:23:05 05 Bash: git status`;
- **оценка выше порога** — авто-подтверждения нет, дальше решает штатный механизм. Если появился
  диалог, запрос приходит в Telegram с пометкой `⚠️ Риск NN/99 — причина`;
- **оценку получить не удалось** (relay недоступен, не настроен `score_url`, LLM не ответила) —
  вызов **не** подтверждается автоматически. «Нет мнения» никогда не трактуется как «безопасно».

Отдельный случай: команда выше порога, которая разрешена списком в `settings.json`, выполнится
молча и без диалога. Такие вызовы приходят отдельным сообщением постфактум:
`⚠️ выполнено без подтверждения (разрешено списком в settings.json)` с оценкой риска — чтобы
рискованное не проскочило незамеченным.

Плата за режим: оценка считается синхронно и задерживает **каждый** вызов инструмента примерно
на секунду, а текст команды уходит во внешнюю LLM — в том числе для вызовов, которые allowlist
пропустил бы молча и которые обычно никуда не отправляются. Повторные одинаковые команды берутся
из кэша и не оцениваются заново.

## Команда /score — оценка риска для всех запросов

`/score` включает показ оценки риска **вне** параноидального режима: **Выключить** / **1 час** /
**4 часа** / **8 часов** / **12 часов** / **Навсегда**. Пока включено, каждый запрос разрешения,
дошедший до Telegram, приходит со строкой `⚠️ Риск NN/99 — причина`.

Оценка считается уже после того, как диалог разрешений реально появился, в фоновом процессе:
вызовы, которые allowlist пропускает молча, никуда не отправляются, и ни один вызов инструмента
не ждёт запроса к LLM. Выбор сохраняется на диск и переживает перезапуск relay (в отличие от `/auto`).

## Команда /default_scores — шкала оценки риска

Показывает диапазоны, по которым выставляются оценки (0–19 чтение, 20–39 локальная запись,
40–59 push и конфиги, 60–79 удаление и секреты, 80–99 необратимое и деструктивное). Тот же список
уходит в промпт модели, так что справка не может разойтись с реальной шкалой.

**Область действия — ваш `chat_id`.** С версии 1.2.0 все настройки (`/wait`, `/length`, `/auto`,
`/score`) привязаны к чату: включив авто-режим у себя, вы не влияете на коллег, использующих тот же relay.
Побочный эффект обновления: сохранённое значение `/length` из версий ≤1.1.0 сбрасывается к 3800
один раз (старый формат хранил одно значение на весь relay, и не было способа понять, чьё оно) —
просто выберите нужное заново.

## Обратная связь из Telegram

Свободный текст, написанный **в теме конкретного проекта** (не команда, не нажатие кнопки), ставится в очередь и подхватывается автоматически при следующем сообщении, отправленном в соответствующей VS Code-сессии — доставка не мгновенная (в Claude Code нет фонового/периодического хука), а «на следующее обращение к сессии».

## Важно

- Если файл секретов не найден — хуки тихо ничего не делают (fail-open), обычный локальный диалог разрешений работает как всегда.
- Общий `relay_secret` технически даёт доступ к любому `chat_id` на этом relay, не только своему — модель доверия рассчитана на небольшую доверенную команду.
- Хук `PreToolUse` запускает фоновый watcher-процесс на каждый вызов Bash/PowerShell/Write/Edit/… — для авто-одобренных (allowlisted) вызовов он ничего не шлёт в Telegram, а живёт до ~15с и молча завершается. На активной сессии это заметный поток короткоживущих процессов — осознанная цена за гонку с нативным диалогом.
- Отключить полностью на время: `$env:TG_APPROVE_OFF = '1'` перед запуском Claude Code / VS Code.
