# Axapta Modification Comments

В проектах внедрения Microsoft Dynamics AX (компания ALK / коды проектов вида `ALK_DEVAX...`) **не используется** система версионирования (нет git/TFS). Поэтому каждая модификация существующего кода и любой новый объект (класс, метод, таблица, форма, EDT, меню и т.д.) маркируется специальными комментариями. Это критически важная конвенция — без неё невозможно отследить, кто и зачем менял код.

## Когда применять навык

Применяй ВСЕГДА, когда:
- редактируешь существующий метод X++ (`.xpo`, `.xpp`)
- добавляешь новый метод, поле, член класса, таблицу, форму, ресурс
- комментируешь/выпиливаешь старый код (он остаётся в исходнике, обёрнутый в `/* ... */`)
- меняешь свойства объектов AOT
- любые другие модификации в проектах с префиксом `ALK_DEV*`

## Параметры модификации

Для маркировки нужны 5 параметров:

| Параметр | Пример |
|----------|--------|
| `<Код проекта>` | `ALK_DEVAX12` |
| `<Код модификации>` | `DAX_012234` |
| `<Описание модификации>` | `[MON] Доработать отчет по расписанию дежурств` |
| `<Текущая дата>` | формат `DD.MM.YYYY`, например `08.05.2026` |
| `<Ник пользователя>` | например `akaz` |

## Нормализация кода модификации

Код модификации принимается в любом виде (`DAX_11233`, `DAX-11233`, `DAX-0011233`, `DAX_012455`).
**Немедленно при получении** нормализовать по алгоритму:

1. Разобрать по шаблону `([A-Za-z]+)[_-]0*(\d+)` → получить `PREFIX` и целое число `N` (без ведущих нулей).
2. **Для хранения** (memory, config): `{PREFIX}-{N}` — дефис, без ведущих нулей. Пример: `DAX-12455`.
3. **Для комментариев в коде**: `{PREFIX}_{N:06d}` — подчёркивание, 6 цифр с ведущими нулями. Пример: `DAX_012455`.

| Ввод | settings / memory | В комментарии |
|------|-------------------|---------------|
| `DAX_11233` / `DAX-11233` / `DAX-0011233` | `DAX-11233` | `DAX_011233` |
| `DAX_012455` / `DAX-12455` / `DAX_12455`  | `DAX-12455` | `DAX_012455` |

**Не сохраняй** ненормализованную форму. Если в memory уже лежит `DAX_012455` (старый формат) — при следующем сохранении перезаписать на `DAX-12455`.

**Текущую дату** всегда брать из системного контекста (поле `currentDate` в системных напоминаниях), приводя к виду `DD.MM.YYYY`. Не используй относительные даты («сегодня», «вчера»).

## Сбор параметров

Перед первым проставлением комментария в сессии — если параметры неизвестны или не закреплены в memory — спроси пользователя через `AskUserQuestion`. Один вопрос на параметр (или один сводный с предзаполненными значениями из memory). Сохранённые в memory значения проекта/ника обычно стабильны между задачами — их предлагай по умолчанию. Код модификации и описание уникальны для каждой задачи — спрашивай явно.

Сразу после получения кода модификации **применить нормализацию** (см. §«Нормализация кода модификации»): хранить в canonical-settings форме (`PREFIX-N`), во всех строках комментариев использовать canonical-comment форму (`PREFIX_NNNNNN`).

После получения параметров **сразу сохрани** их в memory как `project` (текущий тикет: код в canonical-settings форме, описание, дата начала) и в `user` (ник + текущий проект ALK_DEVAX...) — чтобы не переспрашивать в той же или соседней задаче.

## Шаблоны комментариев

### 1. Однострочный — заголовок нового метода/класса/таблицы

```xpp
//ALK_DEVAX12, DAX_012234, [MON] Доработать отчет по расписанию дежурств, 08.05.2026, akaz
class SomeNewClass extends SomeBase
{
    // ...
}
```

или для отдельного метода:

```xpp
//ALK_DEVAX12, DAX_012234, [MON] Доработать отчет по расписанию дежурств, 08.05.2026, akaz
public void newMethod()
{
    // ...
}
```

### 2. Многострочный блок — добавили новые строки внутри существующего метода

```xpp
// + ALK_DEVAX12, DAX_012234, [MON] Доработать отчет по расписанию дежурств, 08.05.2026, akaz -->
new code line 1;
new code line 2;
// - ALK_DEVAX12, DAX_012234, [MON] Доработать отчет по расписанию дежурств, 08.05.2026, akaz <--
```

Открывающий маркер: `// + <параметры> -->` (стрелка `-->`).
Закрывающий маркер: `// - <параметры> <--` (стрелка `<--`).

### 3. Замена существующего кода (старый сохраняется в `/* ... */`)

```xpp
// + ALK_DEVAX12, DAX_012234, [MON] Доработать отчет по расписанию дежурств, 08.05.2026, akaz -->
/*
//ALK_DEVAX12, DAX_007040, ISH: Прошлая модификация, 15.11.2023, rusl
final class SomeClass extends SomeBase
*/
class SomeClass extends SomeBase
// - ALK_DEVAX12, DAX_012234, [MON] Доработать отчет по расписанию дежурств, 08.05.2026, akaz <--
```

Внутри `/* ... */` — старая строка вместе с её прежним маркером модификации (если был). Новая строка идёт после `*/` и до закрывающего маркера.

### 4. Удаление кода (полностью убрать старое)

```xpp
// + ALK_DEVAX12, DAX_012234, [MON] Доработать отчет по расписанию дежурств, 08.05.2026, akaz -->
/*
old code that we are removing;
*/
// - ALK_DEVAX12, DAX_012234, [MON] Доработать отчет по расписанию дежурств, 08.05.2026, akaz <--
```

Старый код всегда сохраняется в `/* ... */`, никогда не удаляется без следа.

## Где проставлять комментарии в `.xpo`

В выгруженных X++ исходниках (`*.xpo`) каждая строка кода метода предварена символом `#`. Комментарий-маркер ставится с таким же `#` и идёт между строк кода метода. Пример:

```
SOURCE #classDeclaration
  #//ALK_DEVAX12, DAX_012234, [MON] Доработать отчет по расписанию дежурств, 08.05.2026, akaz
  #class SomeClass extends Base
  #{
  #    // + ALK_DEVAX12, DAX_012234, [MON] Доработать отчет по расписанию дежурств, 08.05.2026, akaz -->
  #    int newField;
  #    // - ALK_DEVAX12, DAX_012234, [MON] Доработать отчет по расписанию дежурств, 08.05.2026, akaz <--
  #}
ENDSOURCE
```

## Свойство `Origin` для новых объектов, создаваемых с нуля

Реальный AOS Export всегда проставляет в `PROPERTIES` блоке объекта `Origin #{GUID}` — это
внутренний AX-сгенерированный идентификатор. Когда `.xpo` пишется вручную (`Write`/`Edit`), а
не выгружен из настоящей Axapta (типичный случай для нового проекта, который ещё не разворачивался
в реальном AOS):

- **Не изобретай GUID.** GUID состоит СТРОГО из hex-символов `0-9A-F` — буквы вне этого диапазона
  (например встраивание мнемоники кода модификации: `...-DAX124880001`) делают его невалидным.
  AX откажет при импорте: «Недопустимое значение ... для свойства Origin».
- **Пропускай `Origin` целиком** в `PROPERTIES` нового объекта — оставь только `Name`:
  ```
  PROPERTIES
    Name                #ALK_MyNewClass
  ENDPROPERTIES
  ```
  AX присвоит настоящий Origin сама при первом сохранении/импорте — не нужно ничего подставлять.
- Исключение — GUID `Origin` самого `PRN`-объекта Project при сборке релиза через
  [axapta-project-export](../axapta-project-export/SKILL.md): там GUID генерируется штатно через
  `[guid]::NewGuid()` (валидный случайный hex) и переиспользуется между релизами того же
  проекта — это про сам Project-контейнер, не про тела отдельных объектов.

## Гранулярность маркеров

- Один логический блок изменений = один пара маркеров `// + ... -->` / `// - ... <--`. Не дроби каждую строку отдельно, если они меняются вместе.
- Разные логические блоки в одном методе = отдельные пары маркеров.
- Полностью новый метод = одного однострочного `//<параметры>` перед его сигнатурой достаточно. Не оборачивай дополнительно тело метода маркерами `+/-`.
- Полностью новый класс = однострочный комментарий перед `class X extends Y`. Внутренние члены не нужно дополнительно помечать (раз весь класс новый).

## Проверка перед коммитом

1. Все ли новые строки накрыты маркером? (Если строка добавлена, но без маркера — её невозможно отличить от исходной.)
2. Совпадают ли коды проекта/модификации/описание/дата/ник в открывающем и закрывающем маркере? (Они должны быть идентичны.)
3. Старый код, который заменён, сохранён внутри `/* ... */`?
4. Дата актуальна (брать из системы, не выдумывать)?
5. Код модификации в комментариях написан в canonical-comment форме (`PREFIX_NNNNNN`, ровно 6 цифр с ведущими нулями)? Если нет — исправить через `replace_all`.
6. Достигнут естественный чекпоинт (конец фазы многофазного плана, конец сессии с накопленными новыми/изменёнными .xpo, «работает end-to-end»)? Если да — спроси пользователя через `AskUserQuestion`, не собрать ли снапшот через [axapta-project-export](../axapta-project-export/SKILL.md), даже без явной фразы «закрываем тикет» (см. там раздел «Промежуточные чекпоинты в многофазных проектах»).
7. Новые имена локальных переменных/параметров (НЕ полей) не совпадают с зарезервированными
   словами X++ (см. §«Зарезервированные слова X++»)? `validate-xpo` подсветит частичный набор
   случаев как WARN, но полагаться только на авто-проверку нельзя.

## Workflow при первом редактировании в сессии

0. **Preflight ENV-гейт** (см. [setup](../setup/SKILL.md) §«Preflight-конвенция») — прогони:
   ```powershell
   python "${CLAUDE_PLUGIN_ROOT}/scripts/XPOTools/Modules/config.py"
   ```
   Ненулевой exit code → **остановись**, покажи пользователю ошибки из stderr, направь на
   `/alk-axapta-tools:setup`. Не продолжай работу без полной конфигурации — не спрашивай
   отдельные значения вручную, не подставляй временные дефолты.
1. **Проверь структуру `XPO/`** — обязательны и папка `XPO/` в корне, и AOT-раскладка внутри
   (`XPO/Classes/`, `XPO/Data Dictionary/Tables/`, …), см. [axapta-project-export](../axapta-project-export/SKILL.md) §«Папка XPO/ обязательна». Если структура не такая:
   - `.xpo`-файлы вне `XPO/` (в `src/` и т.п.) или папки `XPO/` нет — **спроси** через `AskUserQuestion`,
     перенести ли их в `XPO/`: перемещение меняет структуру репозитория, не делай его молча;
   - `.xpo` в `XPO/`, но лежат плоско — **спроси**, разложить ли по типам через `organize-xpo`
     (скилл [axapta-project-manage](../axapta-project-manage/SKILL.md));
   - файлов ещё нет (новый проект) — создай пустую `XPO/` перед первым `Write` (ничего не переносится —
     подтверждение не требуется);
   - сообщи о созданной папке / переносе / раскладке одной строкой и продолжай.
2. Прочитай memory: ищи `project_current_modification.md` и `user_alk_profile.md`. Если есть и актуальны — используй их.
3. Если memory нет / устарела / пользователь начал явно новую задачу — спроси через `AskUserQuestion`:
   - Код проекта (по умолчанию из memory, иначе — из `AX_PROJECT_ID`, гарантированно заданной preflight'ом)
   - Код модификации (`DAX_NNNNNN` — принимать в любом виде)
   - Описание (с префиксом типа `[MON]`, `ISH:` и т.п. — спроси, если непонятно)
   - Ник пользователя (по умолчанию из memory, иначе — из `AX_USER_NICK`)
   - Сразу нормализовать полученный код: canonical-settings (`PREFIX-N`) → хранить; canonical-comment (`PREFIX_NNNNNN`) → применять в маркерах.
4. Сохрани новые/обновлённые параметры в memory (типы `project` и `user`). Код модификации хранить в **canonical-settings форме** (дефис, без ведущих нулей).
5. **Прочитай конвенцию именования** (до первого нового идентификатора) — благодаря preflight'у
   на шаге 0 значения уже гарантированно заданы, спрашивать не нужно:
   - Прочитай `AX_OBJECT_PREFIX`/`AX_OBJECT_SUFFIX` (ENV, приоритетно) или `config.local.json`.
   - Запомни на сессию: **новые AOT-объекты** → `UPPER_PREFIX_` (например `ALK_`); **методы/переменные/параметры в существующих** → `lower_prefix_` (например `alk_`).
6. Применяй шаблоны комментариев при каждом `Edit` / `Write`; именуй по двухуровневому правилу из п. 5.

## Конвенция именования новых идентификаторов

Все новые идентификаторы в рамках ALK-модификации именуются с проектным аффиксом. Регистр аффикса зависит от того, **что именно** создаётся:

- **Новый AOT-элемент** (новый объект в дереве AOT — класс, таблица, форма, EDT, перечисление и т.д.): аффикс в **ВЕРХНЕМ** регистре. Пример: `ALK_MyNewClass`, `ALK_MyTable`.
- **Новый метод, переменная или параметр** в существующем стандартном объекте (чьё имя не начинается с `ALK_`): аффикс в **нижнем** регистре. Пример: `alk_buildRelationNotExists`, `alk_notExistsIdSet`, `alk_queryBuildDataSource`.

**Ключевое правило**: если контейнер (класс/таблица/форма) уже является ALK_*-объектом (создан в рамках проекта) — дополнительный аффикс на методы и переменные внутри него **не нужен**.

### Глобальная настройка (ENV или config.local.json)

Источник значений по приоритету: **ENV-переменные** `AX_OBJECT_PREFIX` / `AX_OBJECT_SUFFIX`
(их пишет `/alk-axapta-tools:setup`, переживают обновления плагина) → затем `config.local.json`
(лежит в кэше плагина, стирается при обновлении). Preflight-гейт на шаге 0 workflow уже
гарантирует, что ровно одно из двух полей заполнено — здесь их только читают:

```json
{
  "AX_OBJECT_PREFIX": "alk_",
  "AX_OBJECT_SUFFIX": ""
}
```

| Поле | Значение | Смысл |
|------|----------|-------|
| `AX_OBJECT_PREFIX` | `"alk_"` | Аффикс для методов/переменных/параметров в **существующих** объектах (lowercase + `_`) |
| `AX_OBJECT_SUFFIX` | `""` | Постфикс (альтернатива PREFIX). Пусто = не используется |

Ровно одно из двух полей обязательно заполнено (проверяется preflight'ом, не здесь) —
prefix и suffix никогда не заданы одновременно.

**Деривация аффикса для новых AOT-объектов**: взять `AX_OBJECT_PREFIX` (или, если задан
`AX_OBJECT_SUFFIX` — его), привести к UPPER, убедиться что оканчивается на `_` → `ALK_`.

Итого из одного поля `"alk_"` получается:
- Члены существующих объектов: `alk_` (as-is)
- Новые AOT-объекты: `ALK_` (UPPER)

### Что именовать с аффиксом

| Что | Регистр аффикса | Пример при prefix `alk_`/`ALK_` |
|-----|-----------------|----------------------------------|
| **Новый AOT-элемент** (класс, таблица, форма, EDT, перечисление) | **ВЕРХНИЙ** | `ALK_MyNewClass`, `ALK_MyTable` |
| Новый метод в существующем стандартном объекте | **нижний** | `alk_buildRelationNotExists()` |
| Новое поле в `classDeclaration` существующего объекта | **нижний** | `alk_notExistsIdSet` |
| Локальная переменная внутри метода существующего объекта | **нижний** | `alk_joinMode` |
| **Параметр нового метода** в существующем объекте | **нижний** | `alk_queryBuildDataSource`, `alk_joinMode` |
| Метод/переменная/параметр в своём ALK_*-объекте | без аффикса | `buildPopupMenu()`, `_dictRelation` |

### Зарезервированные слова X++

Список зарезервированных слов X++ (компилятор AX не даст объявить с таким именем
**локальную переменную или параметр метода**) — `${CLAUDE_PLUGIN_ROOT}/scripts/XPOTools/Modules/reserved_words.py`
(`RESERVED_WORDS`, сравнение регистронезависимое). Источник — реальный экспортированный
список зарезервированных слов Axapta, сверенный с официальной документацией Microsoft.

**Правило касается ТОЛЬКО локальных переменных и параметров методов.** Поля
`classDeclaration`/`tableFieldsDeclaration` (члены класса/таблицы) под это ограничение
не подпадают — не сверяй и не переименовывай их по этому списку.

**Перед тем как назвать новую локальную переменную или параметр** — сверься со списком
(особенно для коротких «бытовых» имён: `date`, `count`, `window`, `index`, `int`, `str`,
`guid`, `select`, `join`, `order`, `print`, `client`, `server`, `static`, `this`, `new` —
их легко выбрать не задумываясь, и они реально зарезервированы). Если нужное по смыслу
имя зарезервировано — используй аффикс (`alk_date`, `_date`) или синоним (`recDate`,
`dateValue`).

**`validate-xpo` проверяет это автоматически** (WARN) для параметров методов и локальных
переменных — но только эвристически (regex, не полный парсер X++): локальные переменные
детектируются только в начале тела метода (блок объявлений сразу после `{`, до первого
исполняемого оператора — по конвенции ALK они идут именно так). Ручная сверка при выборе
имени всё равно нужна, авто-проверка — подстраховка, а не замена.

### Применение перед реализацией

**До первого `Edit`/`Write`** с новым идентификатором — аффикс уже гарантированно определён
preflight-гейтом на шаге 0 workflow, здесь только применяют, без переспроса:

1. Определи аффикс по приоритету: **ENV** `AX_OBJECT_PREFIX` / `AX_OBJECT_SUFFIX` → затем `config.local.json` (те же поля).
2. Применяй:
   - новые AOT-объекты → `UPPER(AX_OBJECT_PREFIX или AX_OBJECT_SUFFIX)` → `ALK_`
   - методы/переменные/параметры в существующих → как есть (lowercase) → `alk_`
3. Приоритет при расхождениях: ENV > `config.local.json` > memory (`feedback_alk_naming_convention.md`).

## Anti-patterns — чего НЕ делать

- **Не оборачивай новый метод/класс целиком в блок `// + … -->` / `// - … <--`.**  
  Для полностью нового объекта или метода достаточно однострочного header-комментария прямо перед сигнатурой:  
  ```
  #// ALK_DEVAX12, DAX-12306, …, 25.06.2026, akaz
  #public static void myNewMethod(…)
  ```  
  Блочные маркеры `// + … -->` / `// - … <--` — только для добавления/изменения строк **внутри существующего** метода. `validate-xpo --strict` поймает ошибку.
- Не удаляй старый код «насовсем» — оборачивай в `/* */` внутри маркеров.
- Не забывай однострочный маркер на новых методах (легко пропустить, особенно при создании класса с десятком методов).
- Не используй сегодняшнюю дату «по памяти» — бери из `currentDate` в системном контексте.
- Не путай стрелки: открывающий `-->`, закрывающий `<--`.
- Не используй комментарии вида `// TODO`, `// fix later` вместо маркеров — даже временный код должен быть помечен.
- Не пиши описания на английском, если в memory/проекте принят русский (и наоборот).
- **Не фабрикуй GUID для `Origin`** нового объекта (см. §«Свойство `Origin` для новых объектов»).
  Либо валидный `[guid]::NewGuid()`, либо (для тел объектов, не Project) — вообще не указывай
  `Origin`, оставь только `Name` в `PROPERTIES`.
