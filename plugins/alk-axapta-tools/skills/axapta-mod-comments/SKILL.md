# Axapta Modification Comments

В проектах внедрения Microsoft Dynamics AX (компания ALK / коды проектов вида `ALK_DEVAX...`) **не используется** система версионирования (нет git/TFS). Поэтому каждая модификация существующего кода и любой новый объект (класс, метод, таблица, форма, EDT, меню и т.д.) маркируется специальными комментариями. Это критически важная конвенция — без неё невозможно отследить, кто и зачем менял код.

## Когда применять навык

Применяй ВСЕГДА, когда:
- редактируешь существующий метод X++ (`.xpo`, `.xpp`)
- добавляешь новый метод, поле, член класса, таблицу, форму, ресурс
- комментируешь/выпиливаешь старый код (он остаётся в исходнике, обёрнутый в `/* ... */`)
- меняешь свойства объектов AOT
- любые другие модификации в проектах с префиксом `ALK_DEV*`

## Параметры модификации

Для маркировки нужны 5 параметров:

| Параметр | Пример |
|----------|--------|
| `<Код проекта>` | `ALK_DEVAX12` |
| `<Код модификации>` | `DAX_012234` |
| `<Описание модификации>` | `[MON] Доработать отчет по расписанию дежурств` |
| `<Текущая дата>` | формат `DD.MM.YYYY`, например `08.05.2026` |
| `<Ник пользователя>` | например `akaz` |

**Текущую дату** всегда брать из системного контекста (поле `currentDate` в системных напоминаниях), приводя к виду `DD.MM.YYYY`. Не используй относительные даты («сегодня», «вчера»).

## Сбор параметров

Перед первым проставлением комментария в сессии — если параметры неизвестны или не закреплены в memory — спроси пользователя через `AskUserQuestion`. Один вопрос на параметр (или один сводный с предзаполненными значениями из memory). Сохранённые в memory значения проекта/ника обычно стабильны между задачами — их предлагай по умолчанию. Код модификации и описание уникальны для каждой задачи — спрашивай явно.

После получения параметров **сразу сохрани** их в memory как `project` (текущий тикет: код, описание, дата начала) и в `user` (ник + текущий проект ALK_DEVAX...) — чтобы не переспрашивать в той же или соседней задаче.

## Шаблоны комментариев

### 1. Однострочный — заголовок нового метода/класса/таблицы

```xpp
//ALK_DEVAX12, DAX_012234, [MON] Доработать отчет по расписанию дежурств, 08.05.2026, akaz
class SomeNewClass extends SomeBase
{
    // ...
}
```

или для отдельного метода:

```xpp
//ALK_DEVAX12, DAX_012234, [MON] Доработать отчет по расписанию дежурств, 08.05.2026, akaz
public void newMethod()
{
    // ...
}
```

### 2. Многострочный блок — добавили новые строки внутри существующего метода

```xpp
// + ALK_DEVAX12, DAX_012234, [MON] Доработать отчет по расписанию дежурств, 08.05.2026, akaz -->
new code line 1;
new code line 2;
// - ALK_DEVAX12, DAX_012234, [MON] Доработать отчет по расписанию дежурств, 08.05.2026, akaz <--
```

Открывающий маркер: `// + <параметры> -->` (стрелка `-->`).
Закрывающий маркер: `// - <параметры> <--` (стрелка `<--`).

### 3. Замена существующего кода (старый сохраняется в `/* ... */`)

```xpp
// + ALK_DEVAX12, DAX_012234, [MON] Доработать отчет по расписанию дежурств, 08.05.2026, akaz -->
/*
//ALK_DEVAX12, DAX_007040, ISH: Прошлая модификация, 15.11.2023, rusl
final class SomeClass extends SomeBase
*/
class SomeClass extends SomeBase
// - ALK_DEVAX12, DAX_012234, [MON] Доработать отчет по расписанию дежурств, 08.05.2026, akaz <--
```

Внутри `/* ... */` — старая строка вместе с её прежним маркером модификации (если был). Новая строка идёт после `*/` и до закрывающего маркера.

### 4. Удаление кода (полностью убрать старое)

```xpp
// + ALK_DEVAX12, DAX_012234, [MON] Доработать отчет по расписанию дежурств, 08.05.2026, akaz -->
/*
old code that we are removing;
*/
// - ALK_DEVAX12, DAX_012234, [MON] Доработать отчет по расписанию дежурств, 08.05.2026, akaz <--
```

Старый код всегда сохраняется в `/* ... */`, никогда не удаляется без следа.

## Где проставлять комментарии в `.xpo`

В выгруженных X++ исходниках (`*.xpo`) каждая строка кода метода предварена символом `#`. Комментарий-маркер ставится с таким же `#` и идёт между строк кода метода. Пример:

```
SOURCE #classDeclaration
  #//ALK_DEVAX12, DAX_012234, [MON] Доработать отчет по расписанию дежурств, 08.05.2026, akaz
  #class SomeClass extends Base
  #{
  #    // + ALK_DEVAX12, DAX_012234, [MON] Доработать отчет по расписанию дежурств, 08.05.2026, akaz -->
  #    int newField;
  #    // - ALK_DEVAX12, DAX_012234, [MON] Доработать отчет по расписанию дежурств, 08.05.2026, akaz <--
  #}
ENDSOURCE
```

## Гранулярность маркеров

- Один логический блок изменений = один пара маркеров `// + ... -->` / `// - ... <--`. Не дроби каждую строку отдельно, если они меняются вместе.
- Разные логические блоки в одном методе = отдельные пары маркеров.
- Полностью новый метод = одного однострочного `//<параметры>` перед его сигнатурой достаточно. Не оборачивай дополнительно тело метода маркерами `+/-`.
- Полностью новый класс = однострочный комментарий перед `class X extends Y`. Внутренние члены не нужно дополнительно помечать (раз весь класс новый).

## Проверка перед коммитом

1. Все ли новые строки накрыты маркером? (Если строка добавлена, но без маркера — её невозможно отличить от исходной.)
2. Совпадают ли коды проекта/модификации/описание/дата/ник в открывающем и закрывающем маркере? (Они должны быть идентичны.)
3. Старый код, который заменён, сохранён внутри `/* ... */`?
4. Дата актуальна (брать из системы, не выдумывать)?

## Workflow при первом редактировании в сессии

1. Прочитай memory: ищи `project_current_modification.md` и `user_alk_profile.md`. Если есть и актуальны — используй их.
2. Если memory нет / устарела / пользователь начал явно новую задачу — спроси через `AskUserQuestion`:
   - Код проекта (по умолчанию из memory или `ALK_DEVAX12`)
   - Код модификации (`DAX_NNNNNN`)
   - Описание (с префиксом типа `[MON]`, `ISH:` и т.п. — спроси, если непонятно)
   - Ник пользователя (по умолчанию из memory)
3. Сохрани новые/обновлённые параметры в memory (типы `project` и `user`).
4. Применяй шаблоны при каждом `Edit` / `Write`.

## Anti-patterns — чего НЕ делать

- **Не оборачивай новый метод/класс целиком в блок `// + … -->` / `// - … <--`.**  
  Для полностью нового объекта или метода достаточно однострочного header-комментария прямо перед сигнатурой:  
  ```
  #// ALK_DEVAX12, DAX-12306, …, 25.06.2026, akaz
  #public static void myNewMethod(…)
  ```  
  Блочные маркеры `// + … -->` / `// - … <--` — только для добавления/изменения строк **внутри существующего** метода. `validate-xpo --strict` поймает ошибку.
- Не удаляй старый код «насовсем» — оборачивай в `/* */` внутри маркеров.
- Не забывай однострочный маркер на новых методах (легко пропустить, особенно при создании класса с десятком методов).
- Не используй сегодняшнюю дату «по памяти» — бери из `currentDate` в системном контексте.
- Не путай стрелки: открывающий `-->`, закрывающий `<--`.
- Не используй комментарии вида `// TODO`, `// fix later` вместо маркеров — даже временный код должен быть помечен.
- Не пиши описания на английском, если в memory/проекте принят русский (и наоборот).
